1. Concept & the 3 paths

A run takes one support message and walks it through a graph of typed nodes. An LLM classifies it, a mock tool fetches account context, a branch routes it, one action runs, and a second LLM writes the reply. The whole run is persisted so it can be inspected, retried, or resumed.

billing / bug

label → the matching tool node runs, others skipped, final composes reply. Run completed.

unclear

approval node parks → run awaiting_approval until a human clicks Approve, then final runs.

failure

bad LLM output or a tool error → node failed, downstream blocked; Retry resumes.

3. The API → engine router

Every mutating endpoint does the same shape: change one row, then call step(), then return the refreshed run. That single pattern is what makes execute / retry / approve / reject all "just run the loop again".

flowchart LR A["POST /runs/execute/{graph_id}"] -->|create run + node_runs| S["await step(run_id)"] B["POST /nodes/{id}/retry
(must be failed)"] -->|incr_attempts; failed→pending| S C["POST /nodes/{id}/approve
(must be awaiting_approval)"] -->|awaiting_approval→done; set reply| S D["POST /nodes/{id}/reject"] -->|awaiting_approval→failed; set error| S S --> RESP["_run_response()
run + nodes + logs + edges"] E["GET /runs/{id}"] --> RESP F["GET /graphs/{id}"] --> DEF["latest version {nodes, edges}"]

Preconditions are enforced with 409: you can only retry a failed node, only approve/reject an awaiting_approval one.

4. The heart: step()

In engine/executor.py. Stateless and re-entrant: it re-reads every node from the DB on each pass, so it has no memory between calls. Retry and resume are not special — they just call this again.

async def step(run_id, store):
    edges = (await store.get_definition(run_id))["edges"]   # topology, loaded once
    while True:
        nodes  = await store.get_nodes(run_id)              # fresh every pass
        by_key = {n.node_key: n for n in nodes}
        ready  = [n for n in nodes
                  if n.status == "pending"
                  and all(by_key[d].status in ("done","skipped")   # deps resolved?
                          for d in deps_of(n.node_key, edges))]
        if not ready: break
        for n in ready:
            inp = build_input(n.node_key, edges, by_key)    # pull upstream outputs
            await store.set_input(n.id, inp)
            await store.set_status(n.id, "running")
            try:
                out   = await REGISTRY[n.type](n, inp, store)  # dispatch to handler
                fresh = await store.get_node(n.id)
                if fresh.status == "running":               # handler didn't park/skip
                    if out is not None: await store.set_output(n.id, out)
                    await store.set_status(n.id, "done")
            except Exception as e:
                await store.set_error(n.id, str(e))
                await store.set_status(n.id, "failed")
    await reconcile_run_status(run_id, store)               # derive run-level status

reconcile_run_status: any node failed → run failed; else any awaiting_approval → run awaiting_approval; else all resolved → completed; else running.

Why the if fresh.status == "running" check? A handler can change its own status (approval parks itself as awaiting_approval; branch marks others skipped). The loop only auto-completes a node it left running — it never overrides a handler's decision.

5c. Sequence — retry & idempotency

A tool "crashes" after firing its side-effect but before output is saved. The retry must NOT fire it twice.

6. How dependencies are marked (they aren't stored)

There is no "dependencies" column anywhere. A node's deps are derived from the edge list every time in graph.py:

def deps_of(node_key, edges):
    return [e["source"] for e in edges if e["target"] == node_key]   # incoming edges' sources

A node is ready when status == "pending" AND every dep is in ("done", "skipped"). Treating skipped as resolved is the trick that lets a join node fire off one live path:

graph LR cl[classify done] --> br[branch done] fc[fetch_context done] --> br br -->|label=billing| bi[billing done] br -.->|skipped| bug[bug skipped] br -.->|skipped| ap[approval skipped] bi --> fin[final ready ✔] bug -.-> fin ap -.-> fin

branch_handler is what sets the skips: for each outgoing edge whose condition ≠ label, it calls store.set_status(target, "skipped"). Cycle safety is checked once at seed time with is_dag() (Kahn's algorithm).

7. How data flows into a node (build_input)

Edges carry data, not just order. Each edge names a sourceHandle (a key in the upstream output) and a targetHandle (where to place it in this node's input).

def build_input(node_key, edges, nodes_by_key):
    inp = {}
    for e in edges:
        if e["target"] == node_key:
            up  = nodes_by_key.get(e["source"])
            out = (up.output_json or {}) if up else {}
            inp[e["targetHandle"]] = out.get(e["sourceHandle"])   # skipped upstream → None
    return inp

Example — final's input is assembled from 5 edges, giving the compose-agent full context (losing branches are null):

{ "label": "billing",
  "account": {"id":"ACC-42","plan":"pro"},
  "bug_result": null,
  "billing_result": "Checked your invoice ...",
  "approval_result": null }

10. Non-obvious decisions

  • stateless step() No queue, no worker, no daemon. step() runs inline inside the HTTP request. A run "pauses" simply because nobody is calling step(). This makes retry = resume = "call the loop again".
  • deps from edges Dependencies are never stored — always derived from the edge list. Editing a graph can't corrupt an in-flight run (and the run pins its own graph version).
  • skipped = resolved Branching marks losing paths skipped instead of deleting nodes. Readiness treats skipped like done, so join nodes still fire — and the whole run stays inspectable/recoverable.
  • persist before return A tool saves its output to the DB before the handler returns. Combined with the mock's idempotency-key dedupe, a crash mid-side-effect still yields exactly one effect on retry.
  • schema-as-data An agent node's expected output is JSON in its config, turned into a Pydantic validator at runtime. Change a node's contract by editing the graph, not Python. Validation happens before any downstream node runs.
  • fault knobs in mocks fail_until / crash_after_create live in mocks.py, not handlers. Retry/idempotency tests exercise real code paths — no stubs, no monkeypatching.
